# IS/OS 铁律：OS 仅验证，不可优化

> **铁律等级：ABSOLUTE**  
> 违反此铁律的任何操作，无论结果多么诱人，都必须立即停止并报告。

---

## 1. 核心原则

**In-Sample (IS) 是训练场，Out-of-Sample (OS) 是考场。**

| 角色 | IS（样本内） | OS（样本外） |
|------|-------------|-------------|
| **用途** | 参数优化、策略筛选、表达式构造 | **仅**验证 IS 优化结果的泛化稳定性 |
| **允许的操作** | 调参、筛选、排序、迭代改进 | 观察、记录、报告 |
| **禁止的操作** | — | 调参、筛选、排序选优、回溯修改、任何以改善 OS 指标为目的的操作 |
| **决策依据** | IS 指标决定策略是否值得保留 | OS 指标决定策略是否可信（仅做否决用） |

**一句话总结：OS 的唯一合法用途是「一票否决」——告诉你 IS 上好的策略是否真的稳定。OS 不是优化目标。**

---

## 2. 为什么这是铁律

### 2.1 OS 泄露 = 无验证

一旦你在 OS 上做了任何形式的「选择」（选最好的、去掉最差的、调参数让 OS 变好看），OS 就不再是独立的验证集。它变成了第二个 IS，而你失去了所有对未来真实表现的可信估计。

**信息泄露路径（哪怕是无意的）：**

```
IS 上调参 → OS 不好看 → 修改参数 → OS 好看了 → "策略通过"
                                      ↑
                                   这一步就是泄露
```

### 2.2 AI 的天然倾向

AI 在 quant-research 模式下会**本能地**试图改善所有可见指标。当它看到 OS Sharpe = 0.8 时，它会想：
- "能不能换个窗口期让 OS Sharpe 变高？"
- "这个策略 OS 表现不好，让我试试另一个变体"
- "OS 的 decay_ratio 太低了，我微调一下参数"

**这些都是违规行为。** AI 必须被约束为：IS 是工作区，OS 是只读区。

### 2.3 多次 OS 观察的累积泄露

即使你不修改策略，仅仅「看了 OS 结果后决定换一个完全不同的策略方向」，也是一种隐性的 OS 泄露——你在用 OS 信息指导研究方向选择。虽然单次影响微弱，但累积 N 次后等同于对 OS 做了优化。

**缓解措施：** 每次回测仅观察 OS 一次，仅做通过/否决判断，不回溯。如需多轮迭代，必须使用 Walk-Forward 的滚动窗口，而非反复切割同一块 OS。

---

## 3. 禁区行为清单

以下行为在任何情况下都是**严格禁止**的：

| # | 禁止行为 | 典型违规表现 | 严重程度 |
|---|---------|-------------|---------|
| 1 | **对 OS 指标做参数优化** | "OS Sharpe 太低，让我调整窗口期" | 🔴 致命 |
| 2 | **按 OS 指标排序/筛选策略** | "从 10 个变体中选 OS Sharpe 最高的" | 🔴 致命 |
| 3 | **看到 OS 结果后回溯修改策略** | "OS 上 ATR 止损不行，换成固定止损试试" | 🔴 致命 |
| 4 | **用 OS 指标做策略选择** | "两个策略 IS 差不多，选 OS 更好的那个" | 🔴 致命 |
| 5 | **反复切割 OS 观察后调整方向** | "3 次回测 OS 都不好，换个因子方向" | 🟡 严重 |
| 6 | **用 OS decay_ratio 调整截断/中性化** | "OS decay 太高，加个 truncation" | 🟡 严重 |
| 7 | **对 OS 做 walk-forward 后选最优 fold** | "第 3 个 fold 最好，就用它的参数" | 🟡 严重 |
| 8 | **在报告中美化 OS 结果** | "OS Sharpe 不高，但换个起算日就好了" | 🟠 警告 |

## 4. 正确做法

### 4.1 标准流程

```
1. 在 IS 上开发和优化策略
   → IS Sharpe ≥ 阈值? 否 → 废弃或继续优化（在 IS 上）
   → 是 → 进入步骤 2

2. 在 OS 上验证（仅一次，仅观察）
   → OS decay_ratio ≥ 0.5? 否 → 策略不可信，记录失败，不再修改
   → 是 → 策略通过验证

3. 报告结果
   → 同时报告 IS 和 OS 指标，不做任何筛选
```

### 4.2 Walk-Forward 验证（多 fold）

```
对每个 fold:
  - 在该 fold 的训练窗口（IS）上优化
  - 在该 fold 的测试窗口（OS）上仅评估
  - 记录 OS 指标

汇总:
  - 报告所有 fold 的 OS 指标均值/中位数
  - 不选择「最好的 fold」，不做任何基于 OS 的筛选
  - 如果某些 fold OS 极差，记录为风险点，不回溯修改
```

### 4.3 当 OS 表现差时的正确应对

| OS 表现 | 错误反应 | 正确反应 |
|---------|---------|---------|
| OS Sharpe < 0 | "调参数让它变正" | 标记为过拟合，记录失败，换一个完全不同的策略**思路**（非参数微调） |
| OS decay_ratio < 0.3 | "加 truncation/换 neutralization" | 标记为泛化差，记录失败原因 |
| OS 某些 fold 很差 | "去掉差的 fold" | 报告不稳定性，策略不可信 |
| OS 表现平庸 | "微调一下参数应该能好" | 接受现实：策略在 IS 上不够鲁棒 |

**关键区别：** 「换一个完全不同的策略思路」（比如从均线交叉换到动量因子）是允许的——这是探索，不是对 OS 的优化。「微调参数让 OS 变好」是违规——这是对 OS 的泄露。

## 5. 违规判定标准

**判定规则：任何操作，如果其决策依据中包含了 OS 指标，即为违规。**

```
合规: 决策依据 = IS 指标 + 领域知识 + 经济逻辑
违规: 决策依据 = IS 指标 + OS 指标 + ...
                                    ↑
                              不允许存在
```

**灰色地带处理：**
- 如果你不确定某个操作是否违规，**默认视为违规**，先记录再讨论
- 「我已经知道 OS 不好了，所以我想改进策略」→ 违规（你在用 OS 信息驱动改进）
- 「我基于领域知识改进了策略，恰好 OS 也变好了」→ 需要证明改进动机来自 IS 或领域知识，且改进在 OS 验证之前就已确定

## 6. AI 行为约束（针对 quant-research 模式）

AI 在执行量化研究任务时必须遵守以下额外约束：

1. **OS 是只读的。** AI 不得生成任何以「改善 OS 指标」为目的的代码或操作。
2. **OS 验证是一次性的。** 对同一策略的同一组参数，OS 验证只做一次。如果验证失败，策略标记为失败，不做参数回溯。
3. **策略选择只能基于 IS。** 在多个策略候选中选择时，排序和筛选的依据只能是 IS 指标。OS 仅用于对 IS 选出的最优候选做最终否决。
4. **OS 结果必须如实报告。** 不得以任何方式美化、筛选或选择性展示 OS 结果。
5. **每次回测结束后自检。** AI 必须确认本次迭代中未违反 IS/OS 铁律，并在报告中声明。

---

## 7. 自检清单

每次回测迭代后，AI 必须回答以下问题：

- [ ] 我是否在看到 OS 结果后修改了策略参数？ → 如果是，违规
- [ ] 我是否按 OS 指标对多个策略做了排序或筛选？ → 如果是，违规
- [ ] 我选择的策略是否完全基于 IS 指标？ → 如果否，违规
- [ ] OS 验证是否只做了一次（或 WF 每个 fold 一次）？ → 如果否，违规
- [ ] 我是否如实报告了所有 OS 指标，未做筛选？ → 如果否，违规
